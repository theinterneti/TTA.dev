# Publishing TTA.dev Packages

TTA.dev supports two release strategies. Start with GitHub Releases (no registration required) and graduate to PyPI when ready.

| Strategy | Workflow | Requires |
|----------|----------|----------|
| **GitHub Releases** *(recommended to start)* | `release.yml` | Nothing — just push a tag |
| **PyPI** | `publish.yml` | One-time PyPI trusted publisher setup |

Both are triggered by the same tag format: `packages/{dir}/v{semver}`

---

## Option 1: GitHub Releases (low-key, no PyPI account needed)

Packages are built and uploaded as assets on a GitHub Release. Anyone with access to the repo can install them directly — no PyPI account or registration required.

### How it works

Push a tag → `release.yml` builds the wheel + sdist → creates a GitHub Release with the artifacts attached.

### Install a released package

```bash
# Replace TAG and WHEEL with the actual release tag and wheel filename
uv pip install https://github.com/theinterneti/TTA.dev/releases/download/packages/primitives/v1.3.0/tta_dev_primitives-1.3.0-py3-none-any.whl

# Or with uv add (adds to project dependencies)
uv add https://github.com/theinterneti/TTA.dev/releases/download/packages/primitives/v1.3.0/tta_dev_primitives-1.3.0-py3-none-any.whl
```

The exact URL for each asset is shown in the release notes automatically generated by the workflow.

### No setup required

The `release.yml` workflow only needs `contents: write` permission, which is granted automatically by GitHub Actions. No environment, no secrets, no external registrations.

---

## Option 2: Publishing to PyPI

This guide explains how to publish the TTA.dev platform packages to [PyPI](https://pypi.org/) so they can be installed as standard Python dependencies (e.g., `pip install tta-dev-primitives`).

### Overview

TTA.dev uses [OIDC Trusted Publishing](https://docs.pypi.org/trusted-publishers/) — no API keys needed. GitHub Actions authenticates directly with PyPI using short-lived tokens.

The publish workflow lives at `.github/workflows/publish.yml` and is triggered by pushing a tag.

---

### One-Time Setup (per package)

Each package must be registered as a **Trusted Publisher** on PyPI before its first release.

### Step 1: Create the PyPI project (first release only)

If the package has never been published, you must first register it on PyPI:

1. Go to <https://pypi.org/manage/account/publishing/>
2. Click **"Add a new pending publisher"**
3. Fill in the details:
   - **PyPI Project Name**: the package name (e.g., `tta-dev-primitives`)
   - **Owner**: `theinterneti`
   - **Repository name**: `TTA.dev`
   - **Workflow name**: `publish.yml`
   - **Environment name**: `pypi`
4. Click **"Add"**

Repeat for each package you want to publish.

### Step 2: Create the `pypi` environment in GitHub

The workflow uses a GitHub Actions environment named `pypi` for deployment protection:

1. Go to <https://github.com/theinterneti/TTA.dev/settings/environments>
2. Click **"New environment"** → name it `pypi`
3. Optionally add deployment protection rules (e.g., require a review before publishing)

---

### Publishing a PyPI release

#### Package dependency order

Publish packages in this order to ensure dependencies are available on PyPI:

1. **`tta-dev-primitives`** — no internal deps, publish first
2. **`tta-observability-integration`**, **`universal-agent-context`**, **`tta-dev-integrations`**, **`tta-dolt-primitives`** — depend on primitives
3. **`tta-agent-coordination`** — depends on primitives
4. **`tta-documentation-primitives`**, **`tta-kb-automation`** — depend on primitives + observability

#### Step 1: Update the version

Edit the `version` field in the package's `pyproject.toml`:

```toml
# platform/primitives/pyproject.toml
[project]
version = "1.3.1"  # bump this
```

Commit the change:

```bash
git add platform/primitives/pyproject.toml
git commit -m "chore: bump tta-dev-primitives to 1.3.1"
git push
```

#### Step 2: Push a release tag and trigger the workflow

Tags follow the format: `packages/{dir}/v{semver}`

| Package | Tag example |
|---------|-------------|
| `tta-dev-primitives` | `packages/primitives/v1.3.0` |
| `tta-observability-integration` | `packages/observability/v1.0.0` |
| `universal-agent-context` | `packages/agent-context/v1.0.0` |
| `tta-dev-integrations` | `packages/integrations/v0.1.0` |
| `tta-agent-coordination` | `packages/agent-coordination/v1.0.0` |
| `tta-documentation-primitives` | `packages/documentation/v1.0.0` |
| `tta-kb-automation` | `packages/kb-automation/v1.0.0` |
| `tta-dolt-primitives` | `packages/dolt/v0.1.0` |

```bash
# Example: release tta-dev-primitives v1.3.0
git tag packages/primitives/v1.3.0
git push origin packages/primitives/v1.3.0
```

The tag push triggers the `release.yml` workflow which automatically creates a GitHub Release with the built artifacts.

To also publish to PyPI, go to the [Actions tab](https://github.com/theinterneti/TTA.dev/actions/workflows/publish.yml) and manually trigger `publish.yml` with the tag as input.

#### Step 3: Verify the release

- Check the [Actions tab](https://github.com/theinterneti/TTA.dev/actions/workflows/release.yml) to confirm `release.yml` succeeded
- View the new release on the [Releases page](https://github.com/theinterneti/TTA.dev/releases)
- If you also published to PyPI, verify at e.g. <https://pypi.org/project/tta-dev-primitives/>

## Using TTA.dev packages as dependencies

### From GitHub Releases (no PyPI)

```bash
uv pip install https://github.com/theinterneti/TTA.dev/releases/download/packages/primitives/v1.3.0/tta_dev_primitives-1.3.0-py3-none-any.whl
```

### From PyPI (once published)

```bash
pip install tta-dev-primitives
pip install tta-observability-integration
```

Or in `pyproject.toml`:

```toml
[project]
dependencies = [
    "tta-dev-primitives>=1.3.0",
]
```

## Troubleshooting

### Build artifacts in wrong location

The workflow uses `uv build --out-dir dist` from within the package directory to ensure artifacts land in `platform/{package}/dist/`. If you run `uv build` without `--out-dir dist` from a workspace member, artifacts go to the workspace root `dist/` instead.

### Trusted publisher not configured

If the workflow fails with a 403 error, the PyPI trusted publisher has not been configured. Follow the one-time setup steps above.

### Package not found after publishing

PyPI indexing can take a few minutes. If `pip install` fails immediately after publishing, wait a few minutes and try again.

### Version already exists

PyPI does not allow re-uploading to the same version. If you need to fix a release, bump the version number and publish a new tag.
